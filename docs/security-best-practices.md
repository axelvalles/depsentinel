# depsentinel Security Best Practices Guide

This document turns npm/pnpm supply-chain best practices into practical actions with `depsentinel`.

Goal: help any JS/TS project adopt secure defaults with low friction.

## What depsentinel already enforces (M1)

- `scan`: detects package manager (`npm`, `pnpm`, `yarn`, `bun`), lockfile, workspace hint, and framework hint.
- `scan`: returns `risk_score`, `proposed_diff`, and `remediation_commands`.
- `init`: writes secure baseline files (`depsentinel.*`, `.npmrc`, CI workflow).
- `init --preset expo`: writes `pnpm-workspace.yaml` hardened baseline.
- `ci`: fails (non-zero exit code) when critical policy findings exist.
- Unknown framework: no hard-fail; falls back to universal JS/TS baseline checks.

## Secure defaults generated by `init`

### `.npmrc`

Generated baseline:

```ini
ignore-scripts=true
allow-git=none
min-release-age=3
```

Why it matters:

- `ignore-scripts=true`: blocks lifecycle script execution by default.
- `allow-git=none`: blocks git-based dependency resolution.
- `min-release-age=3`: introduces a cooldown before fresh releases are installable.

### `pnpm-workspace.yaml` (preset `expo`)

Generated baseline:

```yaml
packages:
  - "."

minimumReleaseAge: 43200
trustPolicy: no-downgrade
blockExoticSubdeps: true
strictDepBuilds: true
allowBuilds:
  esbuild: true
  rolldown: true
  unrs-resolver: true
```

Why it matters:

- `minimumReleaseAge`: cooldown window to reduce fresh-compromise risk.
- `trustPolicy: no-downgrade`: blocks trust-regression package versions.
- `blockExoticSubdeps`: blocks transitive git/tarball exotic sources.
- `strictDepBuilds` + `allowBuilds`: converts unreviewed build scripts into hard failures.

## Quick start

```bash
# 1) Diagnose
npx depsentinel scan

# 2) Generate secure baseline files (dry-run by default)
npx depsentinel init --preset expo

# 3) Apply generated files
npx depsentinel init --preset expo --write

# 4) Enforce in CI
npx depsentinel ci --json
```

## Mapping: best practice -> depsentinel action

- Disable lifecycle scripts -> `init` writes `.npmrc` with `ignore-scripts=true`.
- Block git dependencies -> `init` writes `.npmrc` with `allow-git=none`; `scan/ci` report risky patterns.
- Use cooldowns -> `.npmrc` `min-release-age`; `pnpm-workspace.yaml` `minimumReleaseAge`.
- Deterministic installs in CI -> CI workflow uses lockfile-enforcing install command per manager.
- Block exotic transitive sources (pnpm) -> `blockExoticSubdeps: true` baseline.
- Enforce trust regression checks (pnpm) -> `trustPolicy: no-downgrade` baseline.

## Coverage matrix: what is automated vs guided

This section maps the two source posts into concrete `depsentinel` coverage.

### A) Implemented by depsentinel today (M1)

- Disable post-install scripts:
  - `init` writes `.npmrc` with `ignore-scripts=true`.
- Block git-based dependencies:
  - `init` writes `.npmrc` with `allow-git=none`.
- Cooldown for new packages:
  - `init` writes `.npmrc` `min-release-age=3`.
  - Expo preset writes `pnpm-workspace.yaml` `minimumReleaseAge`.
- pnpm trust and exotic source controls:
  - Expo preset writes `trustPolicy: no-downgrade` and `blockExoticSubdeps: true`.
- Deterministic CI installs:
  - Generated workflow installs with manager-specific lockfile-enforcing command.
- CI security gate:
  - `depsentinel ci` fails on critical findings.
- Universal JS/TS behavior:
  - Unknown framework degrades gracefully to baseline checks with diagnostics.

### B) Guided in docs, not fully automated yet

- `npx` hardening (offline workspace mode and no-network execution).
- Lockfile injection controls beyond baseline checks (for example `lockfile-lint` pipeline).
- Package install firewalling (`npq`, `sfw`) and package health workflows.
- Secrets hygiene and local isolation (`.env` references + Dev Containers).
- Maintainer publishing posture (2FA, OIDC trusted publishing, provenance attestations).
- Dependency confusion controls (scoped private packages and registry routing).
- Governance layers (SBOM generation/validation, artifact signing, incident playbooks).

### C) Not in M1 by design (planned M2+)

- `fix` command with broad, policy-driven autofix orchestration.
- `doctor` command with unblock playbooks and remediation diagnosis.
- Live advisory feed ingestion with freshness/TTL/offline fallback policy.
- Strong verify gates for lint/format/coverage thresholds.

## Practical guidance for users right now

Use this split to avoid false assumptions:

1. Treat M1 automation as your baseline guardrail (`scan`, `init`, `ci`).
2. Apply guided controls manually in CI/security docs where automation is pending.
3. Track pending controls as backlog items (recommended under a dedicated M2 change).

## Guided snippets for controls not yet automated

### Harden `npx` usage for tooling

```bash
mkdir -p "$HOME/mcp" && cd "$HOME/mcp"
npm init -y
npm install @modelcontextprotocol/server-filesystem
npx --include-workspace-root --workspace "$HOME/mcp" --no --offline @modelcontextprotocol/server-filesystem /path/to/allowed/directory
```

### Add lockfile linting gate

```json
{
  "scripts": {
    "lint:lockfile": "lockfile-lint --path package-lock.json --type npm --allowed-hosts npm --validate-https"
  }
}
```

### Deterministic install commands by package manager

```bash
npm ci
pnpm install --frozen-lockfile
yarn install --immutable --immutable-cache
bun install --frozen-lockfile
```

### Private scope routing to prevent dependency confusion

```ini
@yourcompany:registry=https://npm.yourcompany.com/
```

## Current boundaries (important)

M1 is intentionally scoped. These are not fully implemented yet:

- Full `fix` command (broad autofix orchestration).
- Full `doctor` command with guided unblock playbooks.
- Live advisory feed ingestion (M1 uses a deterministic provider for reliable tests).
- Advanced governance checks (SBOM/provenance/signature verification pipelines).

## Recommended adoption policy

1. Run `scan` on every PR.
2. Keep `ci` as required status check.
3. Start with strict defaults and use explicit overrides only with reason and expiration.
4. Review `proposed_diff` and `remediation_commands` before applying changes.

## Notes for non-Expo projects

- Use `init` without preset for generic JS/TS baseline.
- If your repo uses `pnpm` and you want the same hardening, create or update `pnpm-workspace.yaml` with the settings above.
- If framework detection is unknown, `depsentinel` still applies universal checks and reports diagnostics instead of breaking.
